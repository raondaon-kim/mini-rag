# 제품 기획 가이드

## PRD (Product Requirements Document) 구조

### 1. 개요
- **제품/기능명**:
- **목적**: 왜 이것을 만드는가?
- **성공 지표**: 어떻게 측정하는가?
- **타임라인**: 목표 출시일

### 2. 배경 및 문제 정의
- 현재 상황 (데이터 기반)
- 사용자 고충 (Pain Point)
- 비즈니스 기회

### 3. 사용자 스토리
```
As a [사용자 역할],
I want to [행동],
So that [목적/가치].

수용 기준:
- Given [조건], When [행동], Then [결과]
```

### 4. 기능 우선순위 (RICE)
| 기능 | Reach | Impact | Confidence | Effort | RICE 점수 |
|------|-------|--------|------------|--------|----------|
| | (사용자 수) | (1-3) | (%) | (인/주) | R×I×C/E |

**Impact**: 3=대규모, 2=높음, 1=중간, 0.5=낮음
**Confidence**: 100%=높음, 80%=중간, 50%=낮음

### 5. 범위 정의
| 포함 (In Scope) | 제외 (Out of Scope) |
|----------------|-------------------|
| | |

### 6. 기술 요구사항
- 플랫폼/호환성
- 성능 요구사항
- 보안/개인정보
- 의존성/통합

### 7. 와이어프레임/플로우
```
[화면 A] → [액션] → [화면 B] → [결과]
```

## MoSCoW 우선순위
- **Must**: 없으면 출시 불가
- **Should**: 중요하지만 우회 가능
- **Could**: 있으면 좋음
- **Won't**: 이번에는 안 함

## 체크리스트
- [ ] 문제가 데이터로 검증되었는가?
- [ ] 사용자 스토리에 수용 기준이 있는가?
- [ ] 범위가 명확히 정의되었는가?
- [ ] 기술적 실현 가능성이 확인되었는가?

## 프로액티브 트리거
- 데이터 없이 가설만으로 기획할 때 → 문제 검증 데이터(사용자 인터뷰, 지표) 요청
- 사용자 스토리에 수용 기준(Acceptance Criteria)이 없을 때 → Given/When/Then 추가
- 범위(Scope) 정의가 없어 scope creep 위험이 있을 때 → In/Out of Scope 표 작성
- 기술 검증(Feasibility)이 안 된 기능을 확정할 때 → 기술 검토 후 MoSCoW 재분류
- 성공 지표(Success Metrics)가 없을 때 → 측정 가능한 KPI + 목표값 설정
