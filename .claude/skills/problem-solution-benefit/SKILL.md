# Problem-Solution-Benefit (PSB)

**출처**: 비즈니스 커뮤니케이션 표준 프레임워크. Locker & Kienzler "Business and Administrative Communication" 등 주요 교재에서 채택.

**적용 대상**: 프로젝트 제안서, 사업 기획서, 예산 요청서, 벤더 제안, 내부 비즈니스 케이스

---

## 구조

### 1. 문제 정의 (Problem)

```
- 현재 고통점: [구체적으로, 수치 포함]
- 영향 받는 대상: [이해관계자 명시]
- 방치 시 비용: [손실 회피 심리 활용 — "하지 않으면 $X 손실"]
- 긴급성: [왜 지금 해야 하는지]
```

### 2. 제안 솔루션 (Solution)

```
- 무엇을: [명확한 범위와 산출물]
- 어떻게: [방법론/접근 방식]
- 타임라인: [단계별 마일스톤]
  Phase 1: [기간] — [산출물]
  Phase 2: [기간] — [산출물]
- 필요 리소스: [인력, 예산, 도구]
- 리스크와 완화: [솔직한 평가]
```

### 3. 기대 효과 (Benefit)

```
- 정량적 성과: [매출 $X, 비용 절감 Y%, 효율 Z시간]
- 정성적 성과: [브랜드, 조직 역량, 전략적 포지셔닝]
- ROI / 회수 기간: [투자 대비 수익률]
- 성공 측정 지표: [KPI — 어떻게 측정할 것인가]
```

### 4. 다음 단계 (Call to Action)

```
- 필요한 의사결정: [무엇을 승인해달라는 것인지]
- 의사결정 기한: [언제까지]
- 승인 시 즉시 실행할 것: [첫 번째 행동]
```

## 핵심 원리

> **손실 회피 > 이득 추구**: "이것을 하면 $X를 벌 수 있다"보다 "하지 않으면 $X를 잃는다"가 2배 강력 (Kahneman & Tversky)

## 품질 체크

- [ ] 문제가 수치로 정량화되었는가?
- [ ] "방치 시 비용"이 명시되었는가?
- [ ] 솔루션이 문제와 직접 연결되는가?
- [ ] ROI 또는 회수 기간이 계산되었는가?
- [ ] Call to Action이 구체적인가? (누가, 무엇을, 언제)

## 프로액티브 트리거

- 문제 정의가 모호하거나 추상적이면 → 수치와 구체적 고통점으로 재정의 유도
- 솔루션이 문제와 직접 연결되지 않으면 → Problem↔Solution 매핑 검증 요청
- 수치 기반 Benefit이 없으면 → ROI, 비용 절감, 매출 증가 등 정량적 효과 추가 요구
- "방치 시 비용"이 빠져 있으면 → 손실 회피 프레이밍 보완 제안
- Call to Action이 "검토 부탁드립니다" 수준이면 → 구체적 승인 항목과 기한으로 강화 권고
- 리스크 평가가 누락되었으면 → 리스크와 완화 방안 섹션 추가 유도
