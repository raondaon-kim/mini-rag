# Amazon PRFAQ (Press Release / FAQ)

**출처**: Jeff Bezos, Amazon (2004). "고객부터 역으로 생각하라(Working Backwards)." 파워포인트를 금지하고 내러티브 메모를 의무화한 아마존의 혁신 방법론.

**적용 대상**: 신규 제품 제안, 신규 기능 기획, 서비스 런칭 문서, 혁신 이니셔티브

---

## 보도자료 섹션 (1페이지)

```
[헤드라인]
제품명 + 고객 혜택을 한 문장으로

[부제]
대상 고객 + 핵심 가치

[요약 문단]
제품과 핵심 혜택을 3-4문장으로 요약

[문제 문단]
이 제품이 해결하는 고객의 고통을 구체적으로 서술

[솔루션 문단]
제품이 어떻게 문제를 해결하는지 설명

[리더 인용문]
"우리가 이것을 만든 이유는 [고객 가치]입니다." — [리더 이름, 직함]

[시작 방법]
고객이 지금 바로 할 수 있는 행동

[고객 인용문]
"[제품]을 사용한 후 [구체적 개선]을 경험했습니다." — [가상 고객명, 역할]
```

## FAQ 섹션 (1-5페이지)

### 외부 FAQ (고객 관점)

```
Q: 이 제품은 누구를 위한 것인가요?
A: [타겟 고객 정의]

Q: 기존 [대안]과 무엇이 다른가요?
A: [차별점]

Q: 가격은 얼마인가요?
A: [가격 모델]

Q: 어떻게 시작하나요?
A: [온보딩 과정]
```

### 내부 FAQ (비즈니스 관점)

```
Q: 시장 규모는?
A: [TAM/SAM/SOM]

Q: 수익 모델은?
A: [비즈니스 모델]

Q: 필요한 투자 규모는?
A: [개발비, 운영비, 인력]

Q: 주요 리스크는?
A: [기술적/시장적/운영적 리스크 + 완화 방안]

Q: 성공 지표는?
A: [KPI + 목표치 + 측정 시점]
```

## 핵심 원리

> **불릿 포인트는 불완전한 사고를 숨긴다.** 완전한 문장으로 내러티브를 작성하면, 논리적 허점이 드러난다. — Jeff Bezos

## 품질 체크

- [ ] 보도자료가 고객 혜택 중심으로 작성되었는가? (기술 중심 X)
- [ ] 고객 인용문이 구체적 개선을 포함하는가?
- [ ] 내부 FAQ가 리스크를 솔직하게 다루는가?
- [ ] 읽는 사람이 "이 제품을 쓰고 싶다"고 느끼는가?

## 프로액티브 트리거

- PR이 내부 관점(기술/조직)으로 작성되어 있으면 → 고객 혜택 중심으로 재작성 유도
- FAQ에 진짜 어려운 질문(리스크, 실패 시나리오)이 없으면 → 도전적 질문 추가 요구
- 고객 인용구가 비현실적이거나 추상적이면 → 구체적 개선 수치를 포함한 인용으로 강화
- 헤드라인이 기능 나열형이면 → 고객 혜택 한 문장으로 압축 권고
- 내부 FAQ에 TAM/SAM/SOM이 없으면 → 시장 규모 데이터 보완 요청
- "시작 방법" 섹션이 모호하면 → 고객이 즉시 실행 가능한 구체적 단계로 개선 유도
