# SPIN Framework

**출처**: Neil Rackham, "SPIN Selling" (1988). 12년간 35,000건 영업 사례 분석에 기반. 역사상 최대 규모의 영업 효과성 실증 연구.

**적용 대상**: 복잡한 B2B 제안서, 컨설팅 제안, 고가 서비스 제안, 설득이 핵심인 문서

---

## 구조

### S — Situation (상황 분석)

```
- 고객/조직의 현재 상태 [사실 기반, 의견 아님]
- 관련 배경과 맥락
- 핵심 지표와 벤치마크
```

### P — Problem (문제 식별)

```
- 핵심 문제 진술 [명시적, 구체적]
- 문제 존재의 증거 [데이터, 사건, 불만]
- 문제의 지속 기간과 범위
```

### I — Implication (파급 효과) ⭐ 핵심 단계

> **이 단계가 성공적 제안서와 실패한 제안서를 가른다.**
> 문제를 제시하는 것만으로는 부족 — 독자가 파급 효과의 무게를 느끼게 해야 한다.

```
- 재무적 영향: "이로 인해 연간 약 $X의 손실이 발생합니다"
- 운영적 영향: "매주 Y시간의 지연이 발생합니다"
- 전략적 영향: "Z의 리스크에 노출되어 있습니다"
- 연쇄 효과: "방치 시 다음 문제로 확대됩니다..."
```

### N — Need-Payoff (해결 가치)

```
- "[문제]를 해결하면 $X를 절감/창출할 수 있습니다"
- 제안 솔루션 [Implication과 직접 연결]
- 기대 ROI와 타임라인
- 왜 이 솔루션인가 (대안 대비)
```

## 핵심 원리

> 단순히 문제를 말하지 말고, **파급 효과를 전개하라**. "서버가 느리다"보다 "서버 지연으로 인해 고객 이탈률이 3% 증가했고, 이는 연간 $2M 매출 손실에 해당한다"가 10배 강력하다.

## 품질 체크

- [ ] Implication이 3가지 이상의 파급 효과를 다루는가?
- [ ] 파급 효과가 수치로 정량화되었는가?
- [ ] Need-Payoff가 Implication과 직접 대응하는가?
- [ ] 독자가 "이건 반드시 해결해야 한다"고 느끼는가?

## 프로액티브 트리거

- Situation만 나열하고 Implication이 부족하면 → 파급 효과(재무/운영/전략) 전개 요구
- Need-Payoff가 제품 기능만 나열하면 → 고객 가치와 ROI 중심으로 재작성 유도
- 고객 언어 대신 기술 용어를 사용하면 → 고객 관점 언어로 번역 권고
- Problem이 주관적 의견이면 → 데이터/사건/불만 등 객관적 증거 보강 요청
- Implication의 연쇄 효과가 단선적이면 → 다차원(재무+운영+전략) 파급 구조로 확장 제안
- 대안 비교 없이 솔루션만 제시하면 → "왜 이 솔루션인가" 근거 추가 유도

## 산출물

- SPIN 4단계 구조의 B2B 제안서/컨설팅 문서 (Markdown/DOCX)
- Implication 연쇄 효과 매핑 (정량적 영향 포함)
- Need-Payoff 대비표 (문제↔솔루션↔ROI 매핑)
- 고객 관점 언어로 작성된 설득 내러티브
