# STAR Framework

**출처**: HR 행동면접(Behavioral Interview) 방법론에서 유래 (1980-90년대). DDI(Development Dimensions International) 등에서 체계화. 사례 기반 보고의 표준 구조로 확장.

**적용 대상**: 사례 연구(Case Study), 프로젝트 완료 보고, 성과 보고서, 포트폴리오, 실적 리뷰

---

## 구조

### S — Situation (상황)

```
- 배경과 맥락: 언제, 어디서, 어떤 조직/프로젝트
- 초기 조건: 시작 시점의 상태, 지표, 제약사항
- 이해관계자: 누가 관여했는가
- 규모: 예산, 인원, 기간, 범위
```

### T — Task (과제)

```
- 달성해야 할 목표: [구체적, 측정 가능]
- 성공 기준: [어떻게 성공을 판단하는가]
- 제약과 도전: [시간, 예산, 기술적 제약]
- 나의/팀의 역할: [구체적 책임 범위]
```

### A — Action (행동)

```
- 취한 구체적 행동: [단계별 나열]
  1단계: [무엇을 했는가]
  2단계: [무엇을 했는가]
  3단계: [무엇을 했는가]
- 의사결정 근거: [왜 이 방법을 선택했는가]
- 난관과 극복: [문제 발생 시 어떻게 대응했는가]
- 사용한 도구/방법론: [기술, 프레임워크, 프로세스]
```

### R — Result (결과)

```
- 정량적 성과: [수치 — Before vs After]
  - 매출: $X → $Y (Z% 증가)
  - 비용: $A → $B (C% 절감)
  - 시간: D시간 → E시간 (F% 단축)
- 정성적 성과: [조직적 변화, 역량 향상, 고객 만족]
- 학습 사항: [무엇을 배웠는가, 다음에 다르게 할 것]
- 파급 효과: [이 성과가 다른 영역에 미친 영향]
```

## 핵심 원리

> **Result가 없는 사례는 사례가 아니다.** 반드시 Before/After 수치 비교로 결과를 증명하라.

## 품질 체크

- [ ] Situation이 독자가 맥락을 완전히 이해할 만큼 충분한가?
- [ ] Task에 성공 기준이 명시되었는가?
- [ ] Action이 구체적 단계로 나열되었는가? (모호하지 않게)
- [ ] Result에 Before/After 수치 비교가 있는가?
- [ ] 학습 사항(Lessons Learned)이 포함되었는가?

## 프로액티브 트리거

- Situation이 너무 장황하면 → 핵심 배경만 남기고 간결하게 압축 권고
- Action에서 "나"의 구체적 기여가 불명확하면 → 개인 역할과 의사결정 근거 명시 요구
- Result에 수치가 없으면 → Before/After 정량적 비교 데이터 추가 요청
- Task에 성공 기준이 빠져 있으면 → 측정 가능한 목표 지표 설정 유도
- Action이 단순 나열이고 의사결정 근거가 없으면 → "왜 이 방법을 선택했는가" 보완 제안
- 학습 사항(Lessons Learned)이 누락되면 → 다음 프로젝트에 적용할 교훈 추가 요구

## 산출물

- STAR 4단계 구조의 사례 연구/성과 보고서 (Markdown/DOCX)
- Before/After 수치 비교 포함 결과 분석
- 프로젝트 완료 보고서 (실행 단계 + 정량적 성과)
- Lessons Learned 포함 포트폴리오 항목
